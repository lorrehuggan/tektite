# Plan: Ends - Terminal Trading Game

> Source PRD: https://github.com/lorrehuggan/tektite/issues/6

## Architectural decisions

Durable decisions that apply across all phases:

- **Architecture**: Pure reducer pattern - `gameReducer(state, action) => state` with zero side effects or randomness
- **Randomness**: All random values (prices, encounters, outcome rolls) generated by TUI layer and passed into actions
- **State management**: SolidJS `createStore` wrapping the reducer via `useGame()` hook
- **Screen routing**: Switch/Match on `state.screen` field: `"title" | "market" | "travel" | "encounter" | "game_over"`
- **Module structure**:
  - `app/game/` - Pure game logic (constants, state, actions, selectors, reducer)
  - `app/lib/` - TUI utilities (use-game hook, random helpers)
  - `app/screens/` - Screen-level components
  - `app/components/` - Reusable UI primitives
- **Testing**: Bun test runner, tests in `app/game/__tests__/`

### Game Constants

- **Drugs**: weed, pills, powder, crack, shrooms (with basePrice and volatility)
- **Cities**: camden, brixton, shoreditch, soho, peckham, hackney
- **Starting state**: £2,000 cash, £5,000 debt, 100 health, 100 capacity, day 1, brixton
- **Duration**: 30 days
- **Interest**: 5% per travel
- **Health regen**: +10 per travel (capped at 100)
- **Encounter chance**: 20% per travel
- **Price event chance**: 15% per travel

### Action Contracts

```
NEW_GAME { initialPrices }
BUY { drug, quantity }
SELL { drug, quantity }
PAY_DEBT { amount }
TRAVEL { city, newPrices, encounter, priceEvent }
ENCOUNTER_CHOICE { choice, outcomeRoll }
NAVIGATE { screen }
```

## Phase 1: Core State Machine

**User stories**: 35, 36, 37

### What to build

The foundational game state type, constants, initial state factory, and empty reducer shell. This phase establishes the data model that all subsequent phases build upon. The reducer accepts actions and returns state, but most action handlers simply return unchanged state initially.

Selectors for derived state (net worth, inventory count, inventory value, max affordable, max sellable, price indicator) are implemented and tested.

### Acceptance criteria

- [ ] GameState type defined with all fields (day, maxDays, cash, debt, health, maxHealth, inventory, inventoryCapacity, currentCity, prices, screen, messageLog, activeEncounter)
- [ ] All game constants defined (DRUGS, CITIES, GAME_CONFIG, ENCOUNTER_CONFIG)
- [ ] createInitialState() produces valid starting state given initial prices
- [ ] Action type union defined with all action shapes
- [ ] Reducer handles NEW_GAME action (transitions from title to market)
- [ ] Reducer handles NAVIGATE action (screen transitions)
- [ ] All selectors implemented and passing tests
- [ ] `bun test` runs successfully

## Phase 2: Buy/Sell Logic + Tests

**User stories**: 6, 7, 8, 9, 10, 35

### What to build

Complete BUY and SELL action handlers with full validation. When validation fails (insufficient cash, insufficient capacity, insufficient inventory), the reducer returns unchanged state with an error message added to messageLog. Successful transactions update cash and inventory appropriately.

Priority tests cover all buy/sell scenarios with explicit state inputs.

### Acceptance criteria

- [ ] BUY action increases inventory and decreases cash when valid
- [ ] BUY action fails gracefully with message when cash insufficient
- [ ] BUY action fails gracefully with message when capacity insufficient
- [ ] SELL action decreases inventory and increases cash when valid
- [ ] SELL action fails gracefully with message when selling more than owned
- [ ] Success messages added to messageLog on valid transactions
- [ ] All priority tests for buy/sell passing

## Phase 3: Travel & Day Progression

**User stories**: 13, 14, 15, 16, 17

### What to build

TRAVEL and PAY_DEBT action handlers. Travel advances the day, applies 5% interest to debt, regenerates health (+10 capped at max), updates current city, and replaces prices with new prices from the action. If day exceeds maxDays, screen transitions to game_over.

PAY_DEBT reduces both cash and debt by the specified amount, with validation for insufficient cash.

### Acceptance criteria

- [ ] TRAVEL increments day by 1
- [ ] TRAVEL applies 5% interest to debt (rounded down)
- [ ] TRAVEL regenerates health by 10 (capped at maxHealth)
- [ ] TRAVEL updates currentCity and prices from action
- [ ] TRAVEL transitions to game_over when day > maxDays
- [ ] TRAVEL adds appropriate messages to messageLog
- [ ] PAY_DEBT reduces cash and debt when valid
- [ ] PAY_DEBT fails gracefully when cash insufficient
- [ ] PAY_DEBT handles overpayment (paying more than debt owed)
- [ ] Tests for travel and pay_debt passing

## Phase 5: Market Screen - Display

## Phase 6: Market Screen - Transactions

**User stories**: 11, 12, 16

### What to build

Keyboard interaction for the market screen. Player can navigate the drug list with j/k or arrow keys. Pressing B opens a quantity input modal for buying, S for selling. The modal shows max affordable/sellable and accepts numeric input. Pressing P opens quantity input for paying debt. All modals have Enter to confirm and Esc to cancel.

### Acceptance criteria

- [ ] j/k and arrow keys navigate drug selection
- [ ] B key opens buy modal for selected drug
- [ ] S key opens sell modal for selected drug
- [ ] P key opens pay debt modal
- [ ] Modals show relevant context (price, max quantity, current debt)
- [ ] Numeric input accepted in modals
- [ ] Enter confirms transaction (dispatches action)
- [ ] Esc cancels modal without action
- [ ] T key navigates to travel screen
- [ ] Invalid input shows error feedback
